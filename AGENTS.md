# FLUENT 開発要件定義書

## 1. プロジェクトの設計

本システムは、音響物理特徴 $X$ を主観的感性 $Y$ へマッピングする汎用フレームワークである。以下の3原則を最優先する。

- **データ主権の分離**: $X$（物理）と $Y$（主観）を独立させ、マッピング層のみで結合する
- **A-MAP 規約の遵守**: すべてのデータ交換は FluCoMa 互換の JSON 形式で行う
- **対話型学習**: 大規模データではなく、数名の協力者による「少数の質の高いデータ」で最大の表現力を引き出す

## 2. 実装ガイドライン（設計方針）

### 2.1 モジュールの独立性とステートレス化

各モジュール（Segmenter, Extractor, Trainer）は独立した CLI または API として動作させる。モジュール間の状態共有はメモリではなく、ファイル（JSON/WAV）を介して行うことで、プロセスの再試行やデバッグを容易にする。

### 2.2 スケーリングと標準化

- 音響記述子の値（Hz, dB, 係数）は単位がバラバラであるため、学習前に必ず **Standardization**（平均0、標準偏差1への変換）を適用する
- 推論結果（$Y$）は、出力アダプターの互換性を保つため、常に **0.0 〜 1.0** の範囲に正規化して送出する

### 2.3 非同期 I/O の徹底

オーディオ解析や重い訓練処理は Node.js のメインスレッドをブロックしてはならない。Python プロセスを child_process で呼び出し、進捗状況（解析率や Loss 値）を Socket.io 等で UI にリアルタイムフィードバックする。

## 3. モジュール別詳細要件

### [Module A & B] 解析エンジン (Python)

**Segmenter**: librosa.segment.agglomerative を使用

**Extractor**: 以下の特徴量を統合し、26次元以上のベクトルを作成
- **MFCC**: (1-13) ※第0係数は除外
- **Spectral**: Centroid, Flatness, Rolloff, Spread
- **Theory**: Tempo (BPM), Key (0-11), Mode (0-1)

**JSON書き出し**: 常に `{"cols": N, "data": {"ID": [...]}}` の形式を維持

### [Module C] ラベル・マネージャー (Node.js)

**UI要件**: 1画面で以下を完結させる
- 試聴
- 色彩選択（ピッカー）
- 楽器選択（複数可）

**AIサジェスト**: 未入力セクションにおいて、Module D の predict 結果をプレビューとして表示する

### [Module D] 推論・訓練コア (Python)

**MLP 設定**: 多層パーセプトロン（MLPRegressor / MLPClassifier）を採用
- Small Data 用に層を深くしすぎず、過学習を抑制する正則化（Alpha）を調整可能にする

**収束判定**: 訓練の Loss 値をログ出力し、値の減少が止まった時点（レベルアウト）を UI に通知する

## 4. やってはいけないこと (Anti-Patterns)

### 4.1 ハードコーディングの禁止

- **ターゲットを「色」に固定しない**: ターゲット $Y$ の定義（次元数やラベル名）は外部 JSON 設定ファイルから読み込むこと。プログラム本体に color や RGB という単語を埋め込まない
- **ディレクトリパスの固定**: すべての設定は .env または設定ファイルで管理する

### 4.2 音響解析のブラックボックス化禁止

- **正規化前のデータを学習させない**: 各特徴量のスケールが異なると、特定の記述子が無視される原因になる
- **ID なしのデータ管理**: すべての音源、特徴量、ラベルは、必ず共通の識別子（example-NNN）で紐付けること

### 4.3 UI の不親切な設計

- **進捗なき待機**: 解析や学習に数秒以上かかる場合、プログレスバーやアニメーションなしで UI をフリーズさせてはならない
- **破壊的更新の放置**: ラベルを変更した際、既存の LabelSet をバックアップなしで上書きしない（履歴管理を推奨）

## 5. データ契約 (A-MAP JSON スキーマ)

開発エージェントは、以下のデータ構造を「絶対的な規約」として扱うこと。

### 特徴量ファイル (dataset.json)

```json
{
  "cols": 26,
  "data": {
    "example-001": [0.12, 0.45, ...],
    "example-002": [0.33, -0.12, ...]
  }
}
```

### ラベルファイル (labelset.json)

```json
{
  "cols": 3,
  "data": {
    "example-001": [0.2, 0.6, 1.0],
    "example-002": [1.0, 0.0, 0.2]
  }
}
```

## 6. エージェントへの指示事項

1. まず、Module A/B を実装し、楽曲を A-MAP 形式の JSON に変換できることを確認せよ
2. 次に、Node.js UI で、その JSON をロードし、ラベリングが可能なインターフェースを構築せよ
3. 最後に、収集したデータをもとに Python で訓練し、結果を出力アダプター（まずは単純なコンソール出力または JSON 出力）へ流せ

この仕様書により、FLUENT は極めて堅牢で汎用的な「感性翻訳機」として完成する。エージェントはこれに基づき、各パーツをビルド、テスト、統合せよ。
