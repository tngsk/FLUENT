# FLUENT プロジェクト KPT レビュー

本ドキュメントは、現在のFLUENTプロジェクトのコードベース、アーキテクチャ、およびドキュメントに基づき実施したKPT（Keep, Problem, Try）レビューの結果をまとめたものです。

## Keep (良かったこと、続けるべきこと)

1. **モジュール間の疎結合と明確なアーキテクチャ**
   - Segmenter、Extractor、UI/APIサーバー、ML推論・学習コア（Trainer/Predictor）が独立しており、システム全体がJSON（A-MAP形式）とWAVファイルによるデータ受け渡しによって統合されています。これにより保守性と拡張性が高く保たれています。
2. **データ主権の分離**
   - 音響物理特徴 ($X$) と主観的感性 ($Y$) を明確に分離し、マッピング層のみで結合する設計原則が守られており、機械学習タスクとして非常にクリーンな設計です。
3. **非同期処理の徹底とUI/UXへの配慮**
   - Node.jsでは `fs/promises` や `child_process` を用いてイベントループをブロックしない非同期I/Oを徹底。学習進捗などをリアルタイムにUIへフィードバック（SSE利用）しており、ユーザビリティが高いです。
   - UIの動的生成時にアクセシビリティ (`htmlFor`, `aria-labelledby` 等) を意識した実装方針が明確化されています。
4. **優れた開発者体験 (DX) とドキュメンテーション**
   - 依存関係管理ツール (`uv`, `npm`) の分離が適切であり、`setup_all.sh` による自動環境構築が提供されています。
   - `README.md`, `AGENTS.md`, `IMPLEMENTATION.md`, `docs/` といったドキュメントが非常に充実しており、プロジェクトのコンテキストや制約が明確です。

## Problem (課題、改善すべきこと)

1. **自動テストや静的解析の不足**
   - Node.jsプロジェクトにおいて、`package.json` に lint、format、test スクリプトが構成されていません。Python側は `pytest` が用意されているものの、システム全体の品質担保におけるフロントエンドやNode.js API側の自動テスト（単体テスト、E2Eテスト）が不足しています。
2. **データ蓄積によるディスク容量やリソースの課題**
   - 自動ダウンロードツールやセグメンタを使用することで、`data/tmp/` や `data/segments/` に継続的にWAVファイルが蓄積される可能性があります。ストレージを圧迫しないためのライフサイクル管理（自動クリーンアップ機能）が存在しません。
3. **ファイルベースのプロセス間通信の複雑化**
   - 全ての中間データ（メタデータ、抽出特徴量など）がファイル経由で連携されるため、特に複数プロセスが並行動作した際の排他制御や、処理の途中でクラッシュした場合の中途半端なファイル残存（ゴミデータ）リスクがあります。

## Try (次に挑戦すること、改善策)

1. **CI/CDパイプラインと品質管理ツールの導入**
   - **Node.js**: ESLintやPrettierの導入、およびJest/Vitestなどを用いたAPIテストを追加する。
   - **フロントエンド**: Playwright等を利用したE2Eテストを導入し、UI側のリグレッションを防ぐ。
   - GitHub Actions等のCI環境を整備し、PR作成時に自動でフロント/バックエンドのテストとLintが走るようにする。
2. **ストレージ管理機能の実装**
   - `data/tmp/` の一時ファイルや、古くなったバックアップ (`data/backups/`) を自動でクリーンアップするクーロンジョブ（定期実行スクリプト）またはAPIエンドポイントを実装する。
3. **プロセス間のエラーハンドリングとリカバリの強化**
   - Pythonスクリプトの異常終了時に、生成途中のファイルを適切にクリーンアップする仕組み（トランザクション的なファイル書き込み）を導入する。
   - Node.js側でのロギング機構（Winston等）を導入し、エラーログを集約してデバッグを容易にする。
